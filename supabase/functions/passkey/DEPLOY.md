# نشر دالة الدخول بالبصمة (passkey) — خطوة واحدة لمرة واحدة

هذه الدالة هي خادم التحقق الرسمي من WebAuthn / Passkeys. بدون نشرها لن يعمل
الدخول بالبصمة (لكن الدخول بالبريد وكلمة المرور يظل شغّالاً كالمعتاد دون أي تأثّر).

## المتطلبات
- Supabase CLI مثبّت ومسجّل الدخول (نفس اللي استخدمته لنشر `send-push`).
- `SUPABASE_URL` و `SUPABASE_SERVICE_ROLE_KEY` متوفّران تلقائياً في بيئة Edge
  Functions — لا تحتاج ضبطهما.

## الخطوات

1) شغّل ملف `SETUP.sql` كاملاً في Supabase → SQL Editor (يُنشئ جداول
   `webauthn_credentials` / `webauthn_challenges` / `user_sessions` /
   `auth_audit_log` والدوال والصلاحيات). آمن ومتكرر.

2) اضبط أسرار الدالة (استبدل الدومين لو غيّرته لاحقاً لدومين مخصّص):

```bash
supabase secrets set \
  WEBAUTHN_RP_ID=al-huda-academy.vercel.app \
  WEBAUTHN_ORIGINS=https://al-huda-academy.vercel.app \
  WEBAUTHN_RP_NAME="أكاديمية الهدى"
```

> ملاحظة مهمة: الـ Passkeys مرتبطة بالدومين (RP ID). لو نقلت الموقع لدومين
> مخصّص لاحقاً، سيحتاج المستخدمون إعادة تفعيل البصمة على الدومين الجديد. لدعم
> الاثنين معاً مؤقتاً، ضع الأصلين مفصولين بفاصلة في `WEBAUTHN_ORIGINS` واختر
> `WEBAUTHN_RP_ID` كالدومين الأب المشترك إن أمكن.

3) انشر الدالة:

```bash
supabase functions deploy passkey --no-verify-jwt
```

`--no-verify-jwt` ضروري لأن خطوة الدخول بالبصمة تحدث **قبل** امتلاك جلسة (المستخدم
لسه بيسجّل دخوله). الدالة نفسها تتحقق يدوياً من هوية المستخدم في عمليات التسجيل
(register) عبر رأس Authorization، ومن التوقيع/التحدّي/الأصل في كل العمليات.

## التحقق بعد النشر
- افتح الموقع، سجّل الدخول بالبريد وكلمة المرور، وستظهر نافذة «دخول أسرع بالبصمة؟».
- فعّلها → سيظهر مربع البصمة الرسمي للنظام (Android/Face ID/Windows Hello…).
- سجّل الخروج ثم عد لنفس الجهاز: سيظهر زر «الدخول بالبصمة» فوق نموذج كلمة المرور.
- من القائمة الجانبية → «الأمان وتسجيل الدخول»: تُدار الأجهزة والجلسات وسجل الأمان.

## الأمان
- لا تُخزَّن أي بصمة/وجه/بيانات بيومترية — فقط المفتاح العام ومُعرّف الاعتماد وعدّاد
  التوقيع (منع الاستنساخ). كل عملية بتحدٍّ عشوائي واحد الاستخدام (منع Replay)، مع
  التحقق من Origin و RP ID والتوقيع الرقمي.
- إصدار الجلسة يتم فقط بعد إثبات ملكية المفتاح الخاص تشفيرياً، عبر
  `auth.admin.generateLink` (لا يُرسِل أي بريد — يولّد رمزاً يتحقّق منه العميل فوراً).
